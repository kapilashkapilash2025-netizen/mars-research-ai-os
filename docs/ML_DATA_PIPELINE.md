# Provenance-First AI/ML Data Pipeline

## Objective

The Areograph data pipeline turns traceable source or mission records into deterministic,
training-ready dataset artifacts. It does not train or deploy a model. Its purpose is to establish
the evidence, quality, reproducibility, and human-review boundaries required before model work.

```text
Raw JSONL / verified mission audit
  -> schema and provenance validation
  -> duplicate and range checks
  -> quarantine with reason codes
  -> deterministic identity feature transformation
  -> entity-level train / validation / test split
  -> quality gate
  -> manifest + examples.jsonl + quarantine.jsonl
```

## Required provenance

Every observation requires a source identity, publisher, locator, SHA-256 content digest, licence
identity, processing lineage, and one source classification:

- `source-derived`: traceable to an external published source;
- `synthetic`: intentionally constructed data;
- `inferred`: derived by an identified method;
- `simulated`: generated by a simulation model.

Classification is preserved in every training example. Synthetic, inferred, and simulated records
must never be presented as measurements.

## Label boundary

A non-null supervised label is accepted only when `label_review_status` is `human-reviewed`.
Unreviewed labels are quarantined. When `require_labels` is enabled, the dataset quality gate blocks
training unless at least one reviewed label survives validation. This is an authorization boundary,
not evidence that the label is scientifically correct.

## Determinism and leakage prevention

Canonical JSON plus SHA-256 produces stable example IDs, dataset IDs, configuration hashes, input
hashes, and output hashes. Records are sorted by entity, observation time, and observation ID.
Dataset splitting hashes the entity ID with a versioned salt, ensuring that all observations for an
entity remain in one partition. This prevents direct entity leakage across train, validation, and
test sets.

## Quality gates

The pipeline checks required numeric fields, finite numbers, reviewed ranges, provenance structure,
duplicate canonical observations, accepted-record minimums, quarantine fraction, and optional
label requirements. Invalid records are retained as hash-addressed quarantine entries without
silently modifying their values.

Version 1 deliberately performs identity transformations only. It does not statistically impute
missing data or encode categories because those operations require a task-specific, reviewed model
contract.

## Mission integration

`mission_report_observation()` verifies the audit report content hash before adapting its current
snapshot. The resulting observation remains unlabeled and classified as `simulated`. This lets the
mission engine produce evaluation and research datasets without implying that simulated outcomes
are real Mars telemetry.

## CLI

```powershell
areograph-data-pipeline input.jsonl dataset-output `
  --required-numeric battery_percent `
  --required-numeric wheel_slip `
  --range battery_percent:0:100 `
  --range wheel_slip:0:1
```

The command returns exit code `0` when the quality gate passes and `2` when it blocks the dataset.
Outputs are written atomically as `manifest.json`, `examples.jsonl`, and `quarantine.jsonl`.

## Known limitations and next phase

- No statistical feature fitting, scaling, categorical encoding, model training, or registry exists.
- Licence identifiers are mandatory but policy compatibility is not yet automatically evaluated.
- Local files are the current artifact store; object storage and immutable registry promotion remain.
- Dataset drift, feature drift, and post-deployment performance monitoring remain future work.

The recommended next phase is a reviewed baseline-model laboratory: fit transformations on the
training partition only, compare transparent baseline models, record metrics by scenario and source
classification, generate signed model cards, and require approval before registry promotion.
